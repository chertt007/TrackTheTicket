# Общий флоу приложения (Source of Truth)

Этот документ фиксирует базовую логику продукта.  
Если возникают спорные вопросы по поведению системы, ориентируемся на него как на первичный reference.

## 1. Продуктовый принцип: две независимые проверки цены

Для **каждой подписки** и для **каждого цикла мониторинга** система обязана выполнять два канала:

### Канал A: Fast Source

Назначение:
- быстрая проверка;
- дешевое регулярное получение цены;
- ранний сигнал об изменении;
- грубая валидация рыночной цены.

Реализация может быть через:
- агрегатор;
- API-поставщика;
- сохраненный парсер стабильного источника.

### Канал B: Direct Airline

Назначение:
- получение цены напрямую с сайта авиакомпании;
- сравнение с быстрым источником;
- поиск более выгодного предложения;
- формирование финально более ценного результата для пользователя.

Канал B является ключевым продуктовым преимуществом.

## 2. Основной бизнес-флоу

### 2.1 Создание подписки

Пользователь передает:
- ссылку на билет/поиск;
- условия по багажу;
- частоту отчетов.

Система выполняет:
- извлечение параметров рейса;
- нормализацию параметров;
- определение авиакомпании;
- определение официального сайта авиакомпании;
- создание подписки;
- создание или подготовку стратегии прямого поиска на сайте авиакомпании.

### 2.2 Цикл мониторинга

`Monitoring Orchestrator` запускает последовательность:

1. `Fast Source Check`
2. `Direct Airline Check`
3. `Reconciliation`
4. `Notification`

#### Шаг 1. Fast Source Check

Получаем:
- цену из источника A;
- валюту;
- ссылку/метаданные источника;
- время проверки.

#### Шаг 2. Direct Airline Check

Получаем:
- цену с официального сайта авиакомпании;
- подтверждение соответствия нужному рейсу;
- screenshot;
- детали тарифа.

#### Шаг 3. Reconciliation

Сверяем:
- `price_fast` и `price_direct`;
- условия багажа;
- соответствие `flight_number / time / route`.

#### Шаг 4. Notification

Пользователь получает:
- обе цены;
- разницу;
- лучший вариант;
- источник каждой цены;
- скриншот с сайта авиакомпании.

## 3. Архитектурный принцип

Это **не** fallback-цепочка, а **двухконтурная архитектура проверки**:

- `Dual-Source Monitoring Architecture`

Каждый `CheckJob` содержит:
- обязательный `fast_check_task`;
- обязательный `direct_airline_check_task`;
- агрегирующий `reconcile_and_notify_task`.

## 5. Lifecycle direct-стратегии авиакомпании

Так как direct-site проверка обязательна в каждом цикле, стратегию нужно готовить заранее.

### Этап A: Discovery

Для новой авиакомпании:
- определяем официальный сайт;
- анализируем flow поиска;
- строим стратегию;
- валидируем стратегию.

### Этап B: Operational Use

В обычном мониторинге:
- выполняем сохраненную стратегию;
- не используем AI в runtime-проверке.

### Этап C: Repair

Если сайт изменился:
- фиксируем ошибку выполнения;
- AI ремонтирует стратегию;
- сохраняем новую версию стратегии.

## 6. Роль AI в системе

AI используется для:
- извлечения параметров рейса из сложных ссылок;
- первичного построения стратегии сайта авиакомпании;
- ремонта стратегии при изменении сайта;
- анализа неудачных прогонов.

AI **не** используется для:
- ежедневных обычных проверок;
- Telegram-диалога;
- управления подписками;
- отправки уведомлений;
- расписания;
- сравнения цен.

Ограничение AI runtime-контуром критично для контроля стоимости.

## 7. Набор сервисов и зоны ответственности

1. `Telegram Bot Service`  
   Прием ссылок, вопросы про багаж, частота отчетов, отправка отчетов.

2. `Subscription Service`  
   Создание/изменение/пауза/удаление подписок, хранение параметров мониторинга.

3. `Flight Extraction Service`  
   Извлечение данных из ссылки, нормализация, определение авиакомпании, `flight_signature`.

4. `Airline Discovery Service`  
   Поиск официального домена, маппинг `airline -> official website`, выбор региона/локали.

5. `Fast Price Provider Service`  
   Запрос в быстрый источник и возврат базовой цены для регулярной проверки.

6. `Direct Airline Strategy Service`  
   Хранение стратегий, версионирование, выбор стратегии, валидация.

7. `Browser Automation Service`  
   Запуск Playwright, выполнение стратегии, извлечение цены, screenshot, network logs, HTML snapshot.

8. `AI Strategy Builder/Repair Service`  
   Построение новой стратегии, исправление старой, генерация шагов поиска.

9. `Monitoring Orchestrator`  
   Запуск fast-check и direct-check, ожидание обоих результатов, сравнение, формирование итогового ответа.

10. `Notification Service`  
   Отправка в Telegram пользователю, email разработчику, dev Telegram alert.
